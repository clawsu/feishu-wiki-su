# feishu-wiki-su 开发经验总结

> 适用场景：为 OpenClaw 开发调用外部 HTTP API 的 skill，尤其是飞书系列。

---

## 一、项目结构范式

这套结构经过实践验证，后续同类 skill 直接复用。

```
skill-name/
├── SKILL.md               # OpenClaw skill 入口：YAML frontmatter + 路由逻辑 + HOW 指南
├── README.md              # 给人看的文档：安装、权限、命令参考
├── .gitignore
├── scripts/
│   ├── lib.py             # 共享：HTTP 客户端、鉴权、常量
│   ├── <domain1>.py       # 按业务域拆分，不要揉在一起
│   ├── <domain2>.py
│   └── ...
└── references/
    ├── api.md             # API 速查表（自己维护，核查完再写）
    └── templates.md       # 内容模板/格式规范（如有）
```

**关键原则：**

- `lib.py` 只放共享逻辑（HTTP helper、auth、常量），不放业务
- 每个 script 按业务域划分（wiki 结构 / 文档内容 / 表格 / 电子表格），不按"读/写"划分
- `references/api.md` 是自己维护的速查表，核查完 API 才写进去，不要照抄官方文档
- SKILL.md 是 AI 执行的主要依据，必须覆盖完整的 HOW（step-by-step 执行路径）

## 二、飞书 API 核心坑点

### 2.1 Token 类型混淆（最高频错误）

飞书有三种 token，**绝对不能搞混**：

| Token | 来源 | 用于 |
|-------|------|------|
| `node_token` | wiki URL 或 list-nodes 响应 | wiki 结构操作（移动/重命名） |
| `obj_token` | `get_node` 响应 | 内容操作（读写 docx、bitable、sheet） |
| `app_token` | `obj_token`（当 `obj_type == "bitable"` 时） | 所有 bitable API 调用 |

**正确的 token 链：**
```
wiki URL → node_token → get-node → obj_token → 内容 API
                                              ↓（obj_type == bitable）
                                           app_token → bitable API
```

wiki URL 里的 token 是 `node_token`，**永远不能直接拿来当 `app_token` 或 `document_id` 用**。

---

### 2.2 docx 块操作坑点

**1. 删除块的正确路径**

```
# 错误（会 404 或静默失败）：
DELETE .../blocks/{block_id}/children

# 正确：
DELETE .../blocks/{block_id}/children/batch_delete
```

Body：`{"start_index": 0, "end_index": N}`（左闭右开，和 Python slice 一致）

**2. 修改文档标题**

文档标题（page block）可以通过 PATCH blocks API 修改，`document_id` 和 `block_id` 填同一个值：

```
PATCH /open-apis/docx/v1/documents/{document_id}/blocks/{document_id}
Body: {"update_text_elements": {"elements": [{"text_run": {"content": "新标题"}}]}}
```

官方 FAQ #15 有说明，但很多人以为不支持。

**3. 写入块的字段名**

- 文本块字段名是 `text`，**不是** `paragraph`
- `text_run` 直接作为 key，**没有** `type` 包装层

```json
{
  "block_type": 2,
  "text": {
    "elements": [{"text_run": {"content": "内容"}}],
    "style": {}
  }
}
```

**4. Markdown 不支持的语法（会出错或丢内容）**

| Markdown 语法 | 原因 |
|--------------|------|
| `\| table \|` 表格 | blocks API 报错 1770029 |
| `![img](url)` 图片 | 需要单独 file-upload API |
| `[link](url)` 超链接 | 未在 `_parse_inline` 中实现 |
| 嵌套列表（缩进） | 需要 parent-child block ID，API 复杂 |

---

### 2.3 transfer_owner 参数位置

这个接口的参数分两处放，容易全写到 body 里：

```python
# 正确：
params = {
    "type": "docx",            # 必填 query param
    "need_notification": "false",
    "remove_old_owner": "false",
    "stay_put": "false",
    "old_owner_perm": "view",
}
body = {"member_type": "openid", "member_id": "ou_xxx"}
```

控制选项（4 个布尔）在 query params，身份信息在 body，缺 `type=docx` 会报错。

---

### 2.4 wiki remove-member 参数位置

`member_type` 和 `member_role` 必须在 **request body**，不是 query params：

```python
# 正确：
resp = _api("DELETE",
            f".../spaces/{space_id}/members/{member_id}",
            body={"member_type": "openid", "member_role": "member"})
```

---

### 2.5 bitable 在 wiki 内创建的规则

**wiki 内的 bitable 必须通过 wiki node endpoint 创建：**

```bash
python3 scripts/wiki.py create-node <space_id> --title "表格名" --type bitable
```

直接调用 bitable 创建 endpoint 会报 `1254002`。创建成功后，`obj_token` 才是 `app_token`。

---

### 2.6 bitable 字段操作限制

- `update-field` 是**全量覆盖**，必须同时提供 `field_name` 和 `type`，不支持增量更新
- 主字段（第一个字段/索引字段）**不能删除**，报错 `1254046`
- Select 类型字段需要 `property.options`，格式：`[{"name": "选项名", "color": 0}]`

---

### 2.7 分页处理

飞书所有列表 API 都有分页，`has_more + page_token` 模式。

**标准分页写法（抽成 helper，不要每次手写）：**

```python
def _paginate(tok, path, result_key, extra_params=None):
    params = {"page_size": 100}
    if extra_params:
        params.update(extra_params)
    items = []
    while True:
        resp = _api("GET", path, token=tok, params=params)
        if resp.get("code") != 0:
            _die(f"API error: {resp}")
        data = resp.get("data", {})
        items.extend(data.get(result_key, []))
        if not data.get("has_more"):
            break
        params["page_token"] = data["page_token"]
    return items
```

---

### 2.8 并发写入冲突

bitable 和 sheet 写入都不支持同一张表并发写：

- bitable：相邻批次之间加 `time.sleep(0.5)`，报错码 `1254291`
- sheet：按 serial 顺序写，不做并发

---

### 2.9 权限名称（容易填错）

| 正确 | 错误（常见） |
|------|------------|
| `docx:document` | `docx:document:write` |
| `wiki:wiki` | `wiki:read` |
| `bitable:app` | `bitable:table` |

填错权限名在控制台不会报错，但开放后 API 仍然 403。

---

### 2.10 不在公开 API 的操作

下列操作飞书未开放公开 API，**不要试图实现**：

- wiki 节点删除（delete node）
- 创建 wiki 空间（需要 user_access_token，不适合服务端 app）
- mindnote / slides 内容读写（仅支持 wiki 结构操作）
- sheet 独立的 clear 接口（用写空值代替）

## 三、开发方法论

### 3.1 开发顺序

```
1. 先写 lib.py（HTTP helper + auth）
2. 按业务域逐个写 script，从最简单的开始（wiki → docx → bitable → sheet）
3. 每个 script 写完立即 py_compile 验证语法
4. 用 MCP 核查全部 API，不要凭记忆写路径和参数位置
5. 写 references/api.md（核查完才写，不是开始就写）
6. 写 SKILL.md（最后写，确保代码已稳定）
7. 写 README.md
8. git init → commit → push
```

### 3.2 业务完整性检查框架

每个业务域按 CRUD 逐项核对：

| 操作 | wiki | docx | bitable-table | bitable-field | bitable-record | sheet |
|------|------|------|--------------|--------------|---------------|-------|
| List/Read | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| Get single | ✅ | ✅(blocks) | — | — | ✅ | ✅ |
| Create | ✅ | ✅(write) | ✅ | ✅ | ✅ | ✅(append) |
| Update | ✅(rename) | ✅(rename/write) | — | ✅ | ✅ | ✅(write) |
| Delete | ❌(无API) | ✅(clear/delete-blocks) | ✅ | ✅ | ✅ | ✅(clear=写空) |

**凡是空格或 ❌ 就要搞清楚是「没 API」还是「没实现」**，不能默认跳过。

### 3.3 MCP 核查方法

使用 `mcp__lark-docs__openplatform_developer_document_recall` 工具，**批量查，一次查清楚**：

```
查询粒度：具体到"参数放 query params 还是 body"
不要只查路径，要查：method + path + 必填参数 + 参数位置 + 响应结构
```

**核查时机：写完代码之后集中核查，不要边写边查（效率低）。**

核查顺序建议：
1. 先查所有 DELETE 接口（最容易搞错路径和参数位置）
2. 再查有控制参数的 POST 接口（transfer_owner 这类混合参数的）
3. 最后验证权限名称拼写

### 3.4 常见 bug 类型归类

| Bug 类型 | 典型案例 | 排查方法 |
|---------|---------|---------|
| URL 路径错误 | delete-blocks 缺 `/batch_delete` | MCP 核查路径 |
| 参数位置错误 | transfer_owner 控制项放 body | MCP 查参数位置 |
| Token 类型混淆 | node_token 当 app_token 用 | 代码追踪 token 链 |
| 分页截断 | spaces 只返回前 50 条 | 检查是否有 while + has_more |
| 权限名拼写 | docx:document:write 不存在 | MCP 查 scope 名称 |
| 并发冲突 | bitable 批量写不加延迟 | 加 sleep(0.5) |

## 四、SKILL.md 写法要点

SKILL.md 是 AI 执行的主依据，写法直接影响执行质量。

**必须包含的结构：**

```markdown
---
name: skill-name
description: 触发词描述（越具体越好，包含具体 URL 格式/关键词）
author: ...
metadata: {"openclaw":{"requires":{"env":["KEY1","KEY2"]}}}
---

## SETUP       ← 第一次用的配置步骤
## WHEN        ← 触发条件（越具体越好）
## WHAT        ← 输出物定义
## HOW         ← 执行路径（step by step，含命令）
## REFERENCE   ← 正确/错误示例
## LIMITS      ← 明确说明不支持什么
```

**HOW 部分写法：**

- 用 bash 代码块写命令，不要用自然语言描述步骤
- 先写 Step 0（鉴权错误处理），再写主流程
- 每个分支（bitable/docx/sheet）单独一个 Step，不要混在一起
- 写清楚 token 从哪里来、传给哪个参数

**触发词描述要具体：**

```
# 差：管理飞书文档
# 好：用户提到飞书 wiki/知识库/wiki 空间、提供 feishu.cn/wiki/ URL、
#     想在 wiki 内创建文档/多维表格、管理 wiki 成员
```

## 五、同类项目快速启动 checklist

复制到新项目开始时用：

```
[ ] 确定业务域（wiki / docx / bitable / sheet 各覆盖什么）
[ ] 新建目录结构（scripts/ references/ SKILL.md README.md .gitignore）
[ ] 写 lib.py：_api / _get_token / _out / _die / 常量
[ ] 按业务域逐个写 script，每个写完 py_compile
[ ] 用 MCP 批量核查所有 API（重点：DELETE 路径、混合参数接口、权限名）
[ ] 按 CRUD 矩阵逐项检查业务完整性
[ ] 写 references/api.md（核查后）
[ ] 写 SKILL.md（SETUP / WHEN / WHAT / HOW / REFERENCE / LIMITS）
[ ] 写 README.md
[ ] git init → commit → gh repo create → push
```
